Add analytics_storage consent requirement to all 8 GA4 event tags All 8 GA4 event tags have consentStatus: NOT_SET. This means they bypass the consent gate on the config tag entirely — they fire regardless of visitor consent choice. This is a GDPR/UK GDPR compliance issue. The config tag (Tag 54) is already correctly configured. All event tags must match.
2
Fix banner impression overcounting — redesign Tags 28 and 52 Tag 28 (adrotate_locations) uses an element visibility trigger on .gofollow>img at DOM load. AdRotate places all rotating banners in the DOM simultaneously — so all are counted on page load, including ones that are never shown. Tag 52 (imgSrcEvent) then counts the same banners again via the custom HTML Tag 50.

Recommended fix: Follow the Railway-News v204 approach — replace both with a single Custom HTML tag using IntersectionObserver on .gofollow links, with a deduplication Set. This correctly fires once per banner per page load when the banner actually enters the viewport. One GA4 banner_impression event tag replaces Tags 28 and 52. One GA4 banner_click event tag to be added at the same time (resolves the orphaned ad_rotate_clicks dimension).

Before making such a change: (1) identify which registered dimensions are affected and state the exact date data will stop; (2) confirm whether any Looker Studio report uses that dimension; (3) get explicit confirmation to proceed. 1
All 8 GA4 event tags have consentStatus: NOT_SET Only the config tag (Tag 54) requires analytics_storage. Event tags with NOT_SET are treated as not requiring consent — they bypass the consent gate on the config tag entirely and fire regardless of visitor choice. Under GDPR and UK GDPR this means GA4 event data is being collected without a lawful basis when visitors have not consented. Action: add analytics_storage requirement to all 8 GA4 event tags.
2
Banner impression double-counting — Tags 28 and 52 track the same banners simultaneously Tag 28 (adrotate_locations) fires via element visibility on .gofollow>img (ONCE_PER_ELEMENT at 10%). Tag 52 (imgSrcEvent) fires via Tag 50 (Custom HTML) which loops through all .gofollow > img elements at DOM Ready and pushes a dataLayer event per banner. Both fire for the same banners, creating duplicate data in GA4. Additionally, element visibility fires for all hidden/rotating banners at DOM load — counting banners that are never visible to the user (same overcounting problem as pre-v204 Railway-News). Action: replace Tags 28, 50, 52 with IntersectionObserver approach (Railway-News v204 model).
3
ad_rotate_clicks GA4 custom dimension registered but never populated Registered 19 Jul 2022. No tag in the container fires a click event on banners — no click trigger, no click GA4 tag. This dimension has been empty for approximately 4 years. Action: add banner click tracking (new click trigger + GA4 banner_click tag), or explicitly document as abandoned.
4
client_name GA4 custom dimension registered but never populated Registered 1 Apr 2022. No GA4 event tag sends a client_name parameter. Variable 13 ({{Client Page}}) — which reads data-website from the supplier profile page button — exists in the container but is not wired to any GA4 event. This means client profile page views have never been attributed to specific clients in GA4. Action: build client_page_view tracking using Variable 13.
5
news_category GA4 custom dimension registered but never populated Registered 23 May 2023. Variable 37 ("News Category") reads .col.S-all.M-2.article__tags>h3 text but is wired to no tag. The dimension has been empty since creation. Action: wire Variable 37 to an article_view GA4 tag, or document as abandoned and remove from active reports.
6
published_on GA4 custom dimension registered but never populated Registered 10 Jun 2022. No GA4 tag sends a published_on parameter. Variable 40 ("Post Date New Site") already extracts the post date but is not used in any GA4 tag. Action: build article_view tracking using Variable 40 for published_on.
