Issues & Conflicts

2 critical · 3 warnings ▼
Critical (2)
Warnings (3)
1
Apollo.io (Tag 49) has no GDPR consent gate — fires unconditionally consentStatus: NOT_SET. App ID 66336ab7c350390572a05d4b — same as Railway-News. Apollo.io is a US-based B2B visitor identification tool. Their standard GTM snippet ships without GDPR consent controls. Under GDPR and UK GDPR, processing EU visitor data for B2B identification requires either consent or a documented Legitimate Interests Assessment. As data controller, the consent gate must be added by us.
Fix: Add ad_storage consent requirement to Tag 49 (confirm CMP signals ad_storage first — see Next Steps).
2
All 13 GA4 event tags have consentStatus: NOT_SET — bypass consent gate The config tag (Tag 69) correctly requires analytics_storage. But event tags with NOT_SET are classified as "not requiring consent" and fire regardless of visitor choice — completely bypassing the consent gate. All 13 event tags need analytics_storage added.
1
All 13 GA4 event tags hardcode measurementIdOverride: G-6MVNCW12QM If the GA4 property ever changes, every event tag requires a separate edit. Best practice: remove measurementIdOverride from event tags and inherit from the config tag only. Reduces maintenance risk.
2
console.log() in Tag 54 (AdRotate dynamic Custom HTML) — production noise A console.log() call is present in the AdRotate dynamic custom HTML tag. Outputs debug data to the browser console on every page load. Should be removed before next publish.
3
Tag 56 event name imgSrcEvent — camelCase inconsistency All other GA4 events use snake_case. imgSrcEvent is the only exception. Renaming will break Looker Studio reports or GA4 Explore views that reference this event name — check before changing.

!
Do NOT simply rename the parameters — Looker Studio reports will break If Tags 9 and 12 switch directly from bn_client_name / client_logo_view to client, Looker Studio reports pointing at the old dimensions stop receiving new data. Switching those reports to client makes historical data invisible. Either way you get a reporting gap at the switchover date.

Correct approach: dual-parameter transition (minimum 1 year)

DATE
Old parameters cannot be removed until 1 year after dual-parameter sending starts Client contracts run for 1 year. Looker Studio reporting must cover the full contract period for any active client. Once a start date for dual-parameter sending is confirmed, the earliest safe removal of bn_client_name and client_logo_view is exactly 1 year later.

What dual-parameter sending looks like

TagSends
Tag 9 (supplier_content_impression)bn_client_name = "Bus Company"  +  client = "Bus Company"
Tag 12 (supplier_logo_impression)client_logo_view = "Bus Company"  +  client = "Bus Company"
Tag 68 (client_page_view)client = "Bus Company"  (already correct — no change)

After the 1-year window, remove bn_client_name and client_logo_view from Tags 9 and 12. All reporting runs on client only.
