Critical
1
All 10 GA4 event tags have consentStatus: NOT_SET — bypass consent gate The config tag (Tag 41) correctly requires analytics_storage. But event tags with NOT_SET are classified as "not requiring consent" and fire regardless of visitor choice — completely bypassing the consent gate. The consent management platform becomes ineffective for GA4 tracking.
Fix: Set analytics_storage on all 10 event tags. Done in v1 JSON.
2
Trigger 9 (Client Pages) — /suppliers/ exclusion: EQUALS → CONTAINS Trigger 9 excludes Page Path EQUALS /suppliers/. This only blocks the exact path /suppliers/. All supplier detail pages at /suppliers/company-name/ pass through, so client_page_view (Tag 11) misfires on those pages. The same bug existed in Railway-News (Trigger 225) and was fixed in v204.
Fix: Change condition to CONTAINS — GTM UI change only.
Warnings
3
All 10 GA4 event tags hardcode measurementIdOverride: G-CMEZMGPMVC If the GA4 property ever changes, all 10 tags must be individually updated. Remove from event tags; let them inherit from the config tag (Tag 41).
4
Tag 36 uses min_client_name — non-standard parameter name Tag 11 on this same site uses client. All other containers in the network use client for supplier/client identity. min_client_name likely means "Maritime Industry-News client name" but this abbreviation convention is not used elsewhere. Confirm whether GA4 has a registered custom dimension for min_client_name — if so, apply the 1-year dual-parameter transition before removing it.
5
console.warn in Variables 5 (Post Published Date) and 8 (Author) Both variables contain console.warn('...') in their catch blocks. Unlike console.log in active code paths, this only fires on errors — but it still produces visible browser console output in production. Remove or replace with silent failures (return '').
